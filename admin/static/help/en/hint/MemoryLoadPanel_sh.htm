Memory Loading Panel

This page exhibits the amount of memory paging in/out, database memory usage and the number of times that sessions have to wait for a lock on a buffer in the period between the Start and the End, and lists top memory-consuming OS processes sorted by the column %MEM. The information will be helpful for identifying and tracing performance issues that are happening or had happened on the monitored server.
 

Symptom and Approach

1. CPU utilization will increase as the amount of memory page out increase. If the amount of memory page out is keeping high and the used size of database memory is close to its allocation size, you need to find out what SQL activities have been run (or are running) and how the resource has been taken.
a. Right-click the point and drill down into SQL Point in Time to check what SQLs are/were running and how SQLs are/were running in the interval of the red line.
b. Use the Trend Analysis to view other trend charts for getting more information of resource usage/consumption.
c. If the most memory-consuming process in the OS Process Memory consumption - Real-Time table is a database process, oninit, you may need to diagnose the Top Memory Consuming Session to find out which session is running in the process, how the resource has been taken by the session, and what SQLs are executing etc to get the root cause.
2. If the used size of database memory is close to its allocation size and the buffer waits is keeping high, you may need to take the following actions to find out the root cause.
a. Right-click the point and drill down into SQL Point in Time to check what SQLs are/were running and how SQLs are/were running in the interval of the red line.
b. Use the Trend Analysis to view other trend charts for getting more information of resource usage/consumption.
c. If the most memory-consuming process in the OS Process Memory consumption - Real-Time table is a database process, oninit, you may need to diagnose the Top Memory Consuming Session to find out which session is running in the process, how the resource has been taken by the session, and what SQLs are executing etc to get the root cause.
d. Check what the limitation of database memory allocation is, you may consider increasing it. If there is no limitation for database memory allocation, you need to consider increasing the physical memory in the server.
 

How to Operate

   Trend Chart

 

When you click any place on the diagram, the red line shows as below.

The point-in-time values under the red line on diagrams are displayed in the Legend-Value table as below. They are listed in diagram order from top to bottom. The date and time of the red line shows above the Legend-Value table.

Here we briefly annotate legends appeared in the Legend-Value table.

  • "Page in" and "Page out" are legends for lines representing the memory page in and the memory page out respectively on the "OS Memory Page In/Out" diagram.
  • "DB Virtual Memory Allocation" and "DB Virtual Memory Utilitization" are legends for lines representing the allocated and the used virtual memory respectively on the "DB Memory Allocation/Used(MB)" diagram.
  • "Buffer Waits/sec" is the legend for the line on the "Buffer Waits/sec".

If you would like to view trend charts over a long period of time, please click to pick the start and end dates separately or enter the start and end dates manually. Then click Refresh to redraw diagrams.

   OS Process Memory consumption - Real-Time

 

It is refreshed automatically every couple of seconds to show current and active top most memory-consuming OS processes. You can click a column head to sort the list in ascending/descending order of the selected column.
