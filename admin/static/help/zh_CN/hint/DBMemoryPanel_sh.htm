Database Memory Trend Panel

This page exhibits trend charts of database memory usage, the amount of memory paging in/out, and the cache percentage of buffer read/write between Start and End, and lists top memory-consuming sessions sorted by the column Mem Allocated. These memory usage related information will be helpful for identifying and tracing performance issues that are happening or happened on the monitored server.
 

Symptom and Approach

1. As the amount of memory page out increases, CPU utilization also increases. If the amount of memory page out is keeping high and the used size of database memory is close to its allocation size, you need to find out what SQL activities have been run (or are running) and how the resource has been taken.
a. Right-click the point and drill down into SQL Point in Time to check what SQLs are running (or ran) and how SQLs are running (or ran) in the interval across that point in time.
b. Use the Trend Analysis to view other trend charts for getting more information of dbspace reads/writes and disk I/O activities.
c. Diagnose the most memory-consuming sessions in the Connection Memory Usage table. For how to catch and diagnose symptoms shown in the table, please refer to Top Memory Consuming Session.
2. If the used size of database memory is close to its allocation size, you may need to check what the limitation of database memory allocation is and consider increasing it. If there is no limitation in database memory allocation, you need to consider increasing the physical memory in the server.
3. If the amount of memory page out is high and the buffer read cache rate is low (under 70%), the I/O load increases. You need to what session has a heavy sequential scans SQL and how the SQL is executed.
a. Right-click the point and drill down into SQL Point in Time to check what SQLs are running (or ran) and how SQLs are running (or ran) in the interval across that point in time.
b. Use the Trend Analysis to view other trend charts for getting more information of dbspace reads/writes and disk I/O activities.
c. Use the Top I/O Session to find out which session has heavy sequential scans and/or incremental disk reads, how the resource has been taken by the session, and what SQLs are executing etc to get the root cause.
4. For Connection Memory Usage, please refer to Top Memory Consuming Session to get an idea about how to catch and diagnose symptoms shown in the table.
 

How to Operate

 

When you click or right-click any place on the diagram, the red line shows as below.

The point-in-time values under the red line on diagrams are displayed in the Legend-Value table as below. They are listed in diagram order from top to bottom. The date and time of the red line shows above the Legend-Value table.

Here we briefly annotate legends appeared in the Legend-Value table.

  • "DB Virtual Memory Allocation" and "DB Virtual Memory Utilitization" are legends for lines representing the virtual memory allocation and the virtual memory utilitization respectively on the "DB Memory Allocation/Used(MB)" diagram.
  • "Page in" and "Page out" are legends for lines representing the memory page in and the memory page out respectively on the "OS Memory Page In/Out" diagram.
  • "Buffer Read Cache" and "Buffer Write Cache" are legends for lines on the "Buffer Read/Write Cache Rate " diagram.

If you would like to view trend charts over a long period of time, please click to pick the start and end dates separately or enter the start and end dates manually. Then click Refresh to redraw diagrams.
