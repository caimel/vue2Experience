Locked Sessions

A deadlock occurs when two users hold locks, and each user wants to acquire a lock that the other user owns. Databases are usually designed to detect deadlock situations and prevent them from occurring. In a distributed transaction, deadlocks cannot be detected before they occur. Instead, you can set the DEADLOCK_TIMEOUT configuration parameter that specifies the number of seconds that the database server can wait to acquire a lock before it returns an error. However, certain applications might always be in contention with each other. Examine applications that are producing a large number of deadlocks and try to run them at different times.

You may drill down into Lock Holding Connection and Table in Used to get more information about the holding session and tables referred by the session, or initiate an analysis process through Start Analysis Session to find out the resource utilization and time-consuming of the session and SQLs that are executed in the session during this analysis period, in order to diagnose and analyze the sessions and/or applications further to avoid deadlocks in the future.

And you can drill down into Lock Waiting Connections to view how long the condition exists and how many threads are affected. If necessary, you need to intervene to eliminate the condition.

You can click a column head to sort the list in ascending/descending order of the selected column and click or anytime to refresh information on the page.
